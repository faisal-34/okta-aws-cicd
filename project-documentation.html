✓ Completed AWS Okta Terraform April 2026

Project Scope

This project demonstrates a production-style DevOps pipeline integrating identity management (Okta) with cloud infrastructure (AWS). It covers the full lifecycle: application containerisation, automated CI/CD deployment, and federated AWS Console access via SAML 2.0 SSO.

🚀 CI/CD Automation

Every push to master triggers a 3-job GitHub Actions pipeline: Docker build → Terraform apply → EC2 deploy.

🏗️ Infrastructure as Code

All AWS resources (EC2, IAM, ECR, SAML provider, SSO roles) managed in Terraform with S3 remote state.

🔐 Okta SSO

Users federate into the AWS Console via Okta SAML 2.0 — no AWS credentials required. Role access controlled by Okta group membership.

In Scope

  • Containerised Node.js application (Docker + ECR)
  • GitHub Actions pipeline with 3 sequential jobs
  • EC2 instance provisioned and managed by Terraform
  • IAM roles for EC2 (ECR pull) and Okta SSO (Admin + ReadOnly)
  • AWS IAM SAML Identity Provider (OktaSSOProvider)
  • Okta SAML app with group-based role mapping
  • End-to-end SSO login validated

Out of Scope

  • Okta OIDC integration for the Node.js app itself
  • Custom domain / HTTPS / load balancer
  • Multi-environment (staging/prod) pipeline
  • Automated Okta provisioning in CI (runs manually due to Okta provider timeouts)

VPC & Networking

2

EC2 + Security Group

EC2 instance with Amazon Linux 2023, Docker installed via user data. Security group allows SSH (22) and app traffic (3000). Added associate_public_ip_address = true for accessibility in a non-default VPC.

3

IAM Roles

Two IAM role sets:

  • EC2 Instance Role — okta-cicd-ec2-role with AmazonEC2ContainerRegistryReadOnly — allows EC2 to pull from ECR without static credentials
  • SSO Roles (conditional) — OktaSSOAdminRole (AdministratorAccess) and OktaSSOReadOnlyRole (ReadOnlyAccess) — created only when okta_saml_metadata_xml is provided
4

S3 Remote State

Terraform state stored in okta-cicd-tfstate-493675641194 in us-east-1. Enables the GitHub Actions pipeline to read and update state without local state files.

Phase 2 — CI/CD Pipeline

1

Build Job

Authenticates to ECR with aws ecr get-login-password, builds the Docker image from app/Dockerfile, tags with :latest and the commit SHA, pushes to ECR.

2

Terraform Job

Runs terraform init → terraform plan → terraform apply with -auto-approve. Uses terraform_wrapper: false to prevent output parsing issues. Extracts ec2_public_ip as a job output for the deploy step.

- name: Get EC2 IP
  id: tf-output
  run: echo "ec2_public_ip=$(terraform output -raw ec2_public_ip)" >> $GITHUB_OUTPUT
3

Deploy Job

Uses appleboy/ssh-action to SSH into the EC2 instance using the private key stored as a GitHub secret. Pulls the latest image from ECR and restarts the container on port 3000.

script: |
  aws ecr get-login-password --region us-east-1 | \
    docker login --username AWS --password-stdin 493675641194.dkr.ecr.us-east-1.amazonaws.com
  docker pull 493675641194.dkr.ecr.us-east-1.amazonaws.com/okta-aws-cicd:latest
  docker stop app || true
  docker rm app || true
  docker run -d --name app -p 3000:3000 \
    493675641194.dkr.ecr.us-east-1.amazonaws.com/okta-aws-cicd:latest

Phase 3 — Okta SSO Setup

The Okta Terraform module is run manually (not in CI). The Okta provider caused infinite loops in GitHub Actions due to API polling timeouts. It is decoupled and run once from a local machine.
1

Okta Groups

Two groups created with names matching Okta's AWS group filter pattern:

aws_493675641194_OktaSSOAdminRole
aws_493675641194_OktaSSOReadOnlyRole

The naming convention is critical — Okta extracts accountid and role from these names to build the SAML Role attribute.

2

Okta SAML App

Preconfigured amazon_aws app deployed via okta_app_saml resource. Key settings:

useGroupMapping  = true
joinAllRoles     = false
roleValuePattern = "arn:aws:iam::${accountid}:saml-provider/OktaSSOProvider,
                    arn:aws:iam::${accountid}:role/${role}"
3

AWS SAML Provider

SAML metadata XML extracted from Terraform state and passed to the AWS module via terraform.tfvars (heredoc syntax to preserve XML attribute quotes). Creates aws_iam_saml_provider named OktaSSOProvider.

4

Group Assignment

Both groups assigned to the SAML app via okta_app_group_assignment. The profile block was removed from assignments — the Okta provider v4.20.0 does not support samlRoles as a profile attribute on group assignments.

Bottlenecks & Fixes

1. Okta Terraform provider hung in GitHub Actions

The okta/okta provider made long-polling API calls that exceeded the 15-minute job timeout, causing the pipeline to loop indefinitely and never complete.

Resolution Removed the Okta module entirely from the CI pipeline. The Okta resources are managed separately via a local terraform apply in terraform/okta/. This decoupling also makes sense architecturally — Okta config changes are infrequent and don't need to run on every code push.

2. EC2 created in subnet with no internet routing

Terraform defaulted to the default VPC which had no available subnets in the account. The EC2 instance was unreachable — SSH timed out and the deploy job failed.

Resolution Added explicit vpc_id and subnet_id variables to the AWS module pointing to a subnet with an internet gateway. Also added associate_public_ip_address = true since non-default VPC subnets don't auto-assign public IPs.

3. EC2 couldn't pull from ECR (no IAM role)

The deploy job SSH'd into EC2 and ran aws ecr get-login-password — this failed with a credentials error because the EC2 instance had no IAM instance profile attached.

Resolution Created aws_iam_role.ec2_instance, attached AmazonEC2ContainerRegistryReadOnly, and created an aws_iam_instance_profile. Referenced the profile in the EC2 resource. This is more secure than embedding AWS keys in the deploy script.

4. Deploy job had no EC2 IP — used dynamic lookup returning "None"

The pipeline originally queried the AWS CLI at deploy time to get the EC2 IP. If the instance was still starting or the query returned multiple results, the IP was None and the SSH action failed.

Resolution Changed to capture ec2_public_ip directly from Terraform output in the terraform job and pass it as a job output to the deploy job. This is deterministic and doesn't require a separate API call.

5. okta_app_group_assignment failed — "Property 'samlRoles' not found"

The Okta Terraform provider v4.20.0 removed support for samlRoles as a profile attribute on group assignments. The resource failed to create with a 400 error from the Okta API.

Resolution Removed the profile block entirely from okta_app_group_assignment resources. Role assignment is handled by the group name pattern + useGroupMapping = true on the app, not by per-group profile attributes.

6. SAML metadata XML quotes stripped when passed as CLI variable

Passing the XML metadata as a -var flag on the command line caused PowerShell to strip the double quotes from XML attributes (e.g., encoding="UTF-8" became encoding=UTF-8). AWS IAM rejected the malformed XML with "Could not parse metadata".

Resolution Wrote the XML to a terraform.tfvars file using a heredoc block. This preserves all whitespace and quotes. The tfvars file is gitignored.

7. AWS returned "Invalid SAML response" — no Role attribute

The first successful Okta → AWS redirect failed because the SAML assertion contained no Role attribute. The group names (AWS-Admins, AWS-ReadOnly) didn't match the Okta group filter pattern, so Okta didn't map any role.

Resolution Renamed groups to aws_<accountid>_<rolename> pattern (e.g., aws_493675641194_OktaSSOAdminRole), enabled useGroupMapping = true, and set roleValuePattern to use ${accountid} and ${role} variables extracted from the group name.

8. Okta 400 — conflicting SAML app settings

Setting both identityProviderArn (single-role mode) and useGroupMapping = true (group mapping mode) simultaneously caused Okta to return a 400 on the SSO endpoint. Also, the groupFilter regex used Python-style named groups ((?P<name>)) instead of Okta's syntax ((?{{name}})).

Resolution Removed identityProviderArn from app_settings_json. The provider ARN is handled entirely through roleValuePattern in group-mapping mode. Also corrected the regex syntax to (?{{accountid}}) and (?{{role}}).
